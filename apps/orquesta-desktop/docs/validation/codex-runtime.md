# Codex runtime integration

Orquesta Desktop uses Codex App Server as a local rich-client interface. It does not automate the Codex Desktop window and does not call the general OpenAI API directly.

## Runtime choice

The app discovers an existing Codex runtime in this order:

- `ORQUESTA_CODEX_PATH`
- a standalone `codex.exe` on `PATH`
- common user-level CLI locations
- the newest installed Codex Desktop package under WindowsApps

Orquesta does not bundle a second Codex runtime. The current Windows x64 Codex npm runtime adds roughly 409 MB unpacked, which would break the desktop footprint target before Electron itself is counted.

## Protocol

Core starts `codex app-server --listen stdio://` lazily on the first message or history read. It uses newline-delimited JSON and performs the required `initialize` and `initialized` handshake once per process.

The implementation uses the stable methods generated by Codex 0.144.5:

- `thread/start`
- `thread/resume`
- `thread/read`
- `turn/start`

One coordinator thread is stored per project in Electron's app-owned repository registry. The selected project is used as `cwd`. A message is reported as accepted only after App Server returns both a thread id and a turn id. If a saved thread can no longer be resumed, Core starts a replacement and saves the new id.

The thread uses Codex's `workspace-write` sandbox with approval policy `never`. This avoids adding a second Orquesta security review loop: normal project work can proceed inside the Codex sandbox, while operations that require escalation fail instead of causing a hidden approval dialog.

## UI projection

Core converts App Server notifications into four bounded events: turn started, agent message, turn completed, and turn failed. Renderer receives those events through typed preload methods and shows them as notifications. Raw JSON-RPC messages and process handles never reach Renderer.

Conversation history comes from `thread/read`. Orquesta routing wrappers are stripped before user text is displayed. Messages addressed to a specialist still go through the project coordinator thread, which preserves the coordinator's view of the whole project.

## Shutdown

Closing Orquesta stops file watchers and Core. Core closes App Server stdin first so it can flush its session, waits up to 1.5 seconds, and only then forces termination. Core itself still has the outer bounded shutdown managed by Electron Main.

## Verification

Unit tests cover executable discovery, JSONL framing, initialization, request correlation, process exit, graceful shutdown, thread start and resume, history projection, Core correlation, preload filtering, and IPC input validation.

The Electron runtime test launches the production Renderer, Main, Preload, and Core with a protocol-compatible fake App Server. It sends a Composer message, waits for a real turn acceptance event, receives the coordinator response, and reopens both messages through thread history. The official Codex 0.144.5 binary was also used to generate and inspect the exact TypeScript schema. A paid live model turn is intentionally not part of the automated suite.
