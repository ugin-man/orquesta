# Orquesta 初回セットアップ・オーバーホール設計

作成日: 2026-07-20

この文書は、組織形成とセットアップ内部処理を決めた履歴設計として残している。Desktopの入口、入力画面、Codex認証、書き込み開始点、SetupEngineとの接続については、2026-07-22の[Desktopセットアップ入口・エンジン起動設計](../superpowers/specs/2026-07-22-orquesta-desktop-setup-entry-and-engine-launch-design.md)を現行仕様とする。矛盾する場合は新しい設計を優先し、`session.json`、固定6問、Browser起点を復活させない。

## 目的

初回セットアップを、固定質問と固定チームを配置する半手動の手順から、プロジェクトごとに必要な初期体制を自動構築する製品機能へ変更する。

ユーザーが最初に見る体験は、入力フォームを消化する作業ではなく、自分のプロジェクトを説明し、その説明からOrquestaの組織が組み上がる過程を確認する体験にする。

## 現在の問題

現行の初回セットアップには次の問題がある。

- 専用のセットアップエンジンがなく、統括者が手順書を読みながら状態ファイルを作っている。
- Windowsアプリの初回起動と、プロジェクトごとの初回起動が一つの製品フローとしてつながっていない。
- デスクトップアプリはセットアップ状態を読めない。
- 未初期化プロジェクトは、`agents.json`や`tasks.json`がないため通常プロジェクトとして開けない。
- 利用者連絡係、構想整理係、障害相談係が別々の長期スレッドになり、役割より運用コストの方が大きい。
- 必須質問が固定され、プロジェクト説明を実質的に使えていない。
- 完成マップと専門家候補がOrquesta自身の開発向けテンプレートに固定されている。
- 旧ウェブダッシュボードの起動と確認がセットアップの中心になっている。
- 進捗表示が実処理イベントと結び付いていない。

## 製品境界

セットアップには二つの入口を用意し、プロジェクトが確定した後は同じエンジンへ合流させる。

### Codexからインストールする場合

1. ユーザーが現在のCodexプロジェクトでOrquestaのインストールを依頼する。
2. CodexがWindowsアプリをインストールする。
3. Codexが現在のプロジェクトルートを指定してOrquestaを起動する。
4. アプリは対象フォルダを表示し、ユーザーの確認を待つ。
5. ユーザーが確認すると初期入力へ進む。

プロジェクトを知っているにもかかわらず、空のフォルダ選択画面をもう一度出してはならない。確認画面には、検出したプロジェクトと「このプロジェクトで始める」「別のプロジェクトを選ぶ」を表示する。

### 手動でインストールする場合

1. Windowsインストーラーの完了後にOrquestaを起動する。
2. プロジェクト情報が渡されていなければ、「既存プロジェクトを選ぶ」「新しいプロジェクトを作る」を表示する。
3. プロジェクトが確定したら、Codex経由と同じ初期入力へ進む。

### 導入済み環境で新規プロジェクトを追加する場合

1. ユーザーが新しいプロジェクトを選択または作成する。
2. `.orquesta`が存在しない場合はプロジェクト確認へ進む。
3. 中断状態が存在する場合は新規作成せず再開する。
4. 既存状態が古い場合は通常セットアップではなく移行フローへ分岐する。

## 基礎エージェント

新規プロジェクトで必ず作る長期エージェントは三体とする。

### 統括者

- プロジェクト理解
- 完成マップと初期計画
- 専門家編成
- タスク分解と割り振り
- 成果物の受理
- 最終的なユーザー報告

### 利用者支援係

従来の利用者連絡係、構想整理係、障害相談係を統合する。

- 専門家から届いた質問の重複整理と優先順位付け
- ユーザー回答の保存と暫定解釈
- ユーザータスクの整理
- ユーザー側の作業が必要なエラーの整理
- 方針変更、承認、確認事項の提示
- 回答済み事項と未確認事項の区別

普段は待機状態とし、質問、ユーザー作業、反復エラー、判断候補が発生した場合だけ起動する。

### 管理係

- Orquestaの初回セットアップ
- 設定と診断
- バージョン更新と状態移行
- Codexランタイム接続の確認
- デスクトップ表示と状態監視の修復
- ユーザーがタスクやOrquestaの状態を理解するための案内

プロジェクト成果物の実装や専門判断を担当してはならない。

### 既存プロジェクトの統合

既存の`user-liaison`、`vision-curator`、`error-concierge`は即時削除しない。

- 会話、タスク、レポート、判断履歴を保持する。
- 新しい利用者支援係へ責務を移す。
- 旧エージェントに`superseded_by`と移行日時を記録する。
- 旧エージェントを履歴表示可能な退役状態へ変更する。
- 同じプロジェクトで新旧両方がユーザー質問を生成しないようにする。

## 初期入力

### プロジェクト名

次の順番で初期値を決め、ユーザーが編集できるようにする。

1. Codexから渡された名前
2. プロジェクトのマニフェスト
3. Gitリポジトリ名
4. フォルダ名

新しいフォルダをアプリから作った場合は、そのときの名前を使う。

### プロジェクト説明

入力欄は残す。空欄から書かせるのではなく、次の情報から下書きを作る。

- インストールを依頼したCodex会話
- README
- マニフェスト
- 主要設計書
- 既存の引き継ぎ文書
- ファイル構成

ユーザーは下書きを確認し、間違いだけを直せる。

### 補完質問

固定質問と必須6問は廃止する。

- 質問数は0件から3件まで。
- 回答によって初期計画、専門家構成、最初の作業が変わる場合だけ生成する。
- 一つの画面にまとめる。
- 質問ごとに、なぜ必要かを短く示す。
- 全件を飛ばせる。
- 飛ばされた事項は勝手に確定せず、暫定または未確認として記録する。
- 将来の好みや細部はセットアップを止めず、通常運用中の質問候補へ送る。

### オプションパック

初回セットアップでは表示しない。

内部の能力パックが必要な場合はプロジェクト内容から自動選択し、後から設定画面で確認、変更できるようにする。ユーザーには技術的なパック名ではなく、何ができるかを表示する。

## セットアップ開始確認

初期入力後、状態ファイルとCodexスレッドを作る前に一度だけ確認する。

表示内容:

- 対象プロジェクト
- プロジェクト名
- プロジェクト説明の短い要約
- 必ず作る基礎エージェント三体
- 補完質問の回答件数と未回答件数
- 後から変更できること

主要操作は「初期セットアップを開始」とする。

## 六段階のセットアップエンジン

### フェーズ1: 環境確認

実行内容:

- プロジェクトルートの存在確認
- 書き込み可能性の確認
- Codexランタイムの確認
- 既存状態、中断状態、移行必要性の判定
- セットアップジャーナルの作成

この段階で続行不能な場合だけユーザーを止める。

### フェーズ2: プロジェクト理解

実行内容:

- 初期入力とCodexからの引き継ぎを読む
- README、マニフェスト、主要設計書を読む
- プロジェクト種類、現在段階、技術構成を推定する
- 最初に達成すべき成果を決める
- 推定結果と根拠を保存する

リポジトリ全体を無制限に読み込まない。決定論的なファイル探索で入口を絞り、一回の統合判断へ渡す。

### フェーズ3: 基盤構築

実行内容:

- `.orquesta`状態領域を作る
- 統括者を登録する
- 利用者支援係を登録する
- 管理係を登録する
- 質問、ユーザータスク、エラー、履歴、レポート領域を初期化する
- デスクトップ監視用の状態を用意する

### フェーズ4: 初期計画

実行内容:

- 暫定的な完成条件を作る
- プロジェクト固有の完成マップを作る
- 最初のマイルストーンを決める
- 必要能力を整理する
- 未確認事項を明示する
- 既存資産や外部ツールを探索すべき領域を記録する

初回セットアップ中に無制限なWeb探索は行わない。本格探索は最初の実タスクへ引き継ぐ。

### フェーズ5: 専門家編成

実行内容:

- 最初のマイルストーンに必要な専門家を選ぶ
- 役割、担当範囲、必要資料、除外文脈、禁止事項を決める
- 専門家を状態へ登録する
- 長期Codexスレッドを作る
- 統括者との接続関係を記録する
- 将来必要な専門家を候補として保存する

専門家を作るが、全員へ準備レポートを書かせない。統括者が一度の判断で役割契約を作り、各専門家は最初の仕事が来るまで待機する。

### フェーズ6: 運用開始

実行内容:

- デスクトップ表示と状態監視を接続する
- 組織図、完成マップ、ユーザータスクを反映する
- 最初の作業候補を用意する
- 状態を一度だけ読み直す
- セットアップ完了を記録する

完了確認は状態ファイル、基礎三体、初期専門家、デスクトップ投影の四点に限定する。同じ検証を複数エージェントに繰り返させない。

## 状態機械

セットアップ全体の状態:

- `not_started`
- `collecting_input`
- `ready_to_start`
- `running`
- `waiting_user`
- `blocked`
- `cancelling`
- `cancelled`
- `completed`

各フェーズの状態:

- `queued`
- `running`
- `completed`
- `blocked`
- `skipped`

各アクティビティの状態:

- `queued`
- `running`
- `completed`
- `failed`

デスクトップは進捗を推測しない。セットアップエンジンが書いた現在フェーズ、アクティビティ、時刻、短い証拠だけを表示する。

## 保存契約

新しい正本として`.orquesta/setup/session.json`を追加する。

最低限、次を保存する。

- `version`
- `setup_id`
- `status`
- `entry_route`
- `project_root`
- `started_at`
- `updated_at`
- `completed_at`
- `current_phase_id`
- `phases`
- `activities`
- `project_profile_ref`
- `completion_map_ref`
- `specialist_plan_ref`
- `blocking_issue`
- `resume_token`

`.orquesta/state/events.jsonl`には状態変化の履歴を追記する。`session.json`は現在状態、`events.jsonl`は履歴として使い分ける。

## 中断、再開、取消

- 同じ`setup_id`の処理を二重起動しない。
- 完了したフェーズは再実行せず、未完了フェーズから再開する。
- ファイル書き込みはフェーズ単位で確定する。
- 取消時は完成済み状態を破壊せず、セットアップを`cancelled`として記録する。
- 再開可能な途中状態を残す。
- 作成済みスレッドは削除せず、セットアップ由来の待機状態として記録する。
- 既存プロジェクトファイルは変更しない。

## デスクトップのセットアップ画面

通常のホーム画面とは別の専用画面を使用する。動画は使わず、状態イベントで動くUIを作る。

### 背景

- 実在のパイプオルガンをモチーフにした背景を最初から表示する。
- Orquestaロゴを大きく引き伸ばしたものにはしない。
- パイプオルガンは進捗表示ではなく、Orquestaそのものを表す。
- 背景は暖かい白、紙の質感、淡い銀色、鉛筆線で構成する。
- テキストの可読性を損なわない濃度にする。
- 最終フェーズでは風の流れ、シャッター、淡い青白い光で起動を表す。

### 左側

- 現在のフェーズ名
- 現在行っている処理
- 完了した直近の処理
- 次に行う処理
- 必要な場合だけ時刻
- 「技術的な詳細を表示」

ログ全文を常時表示せず、ユーザーが理解できる短い処理名を出す。

### 右側

- 六段の縦型歯車機構
- 歯車の横に六つのフェーズ名
- 完了、実行中、待機中の明確な差
- 実行中フェーズだけ回転させる
- 完了時にロック機構を動かす
- 次のフェーズへ移るときに軸、ラッチ、歯車で重量感を出す

歯車はパイプオルガンを直接組み立てない。パイプオルガンと歯車は視覚的、意味的に分離する。

### 最終遷移

フェーズ6が完了すると、歯車が順番に停止し、パイプオルガンが起動する。短い完了表示の後、専用画面全体をフェードさせて通常のホーム画面を表示する。

既存ホーム画面の組織図へ部品を移動させる演出は行わない。

## モーション規則

- 動画ファイルで進捗を偽装しない。
- 実際のフェーズイベントで歯車とテキストを更新する。
- 完了したフェーズは再び回転させない。
- 処理が止まった場合、アニメーションも止める。
- `prefers-reduced-motion`では回転と移動を止め、濃度と状態ラベルだけで進捗を伝える。
- 画面の常時点滅は使わない。
- 状態変化の演出は短くし、継続的な背景アニメーションは弱くする。
- 処理が速くても、初回の構築演出は最低約2秒表示する。
- 完了済み処理を待たせるためだけに長い演出を続けない。

## エラー表示

続行不能な場合だけセットアップを止める。

- 左側の現在処理をエラー内容へ置き換える。
- 問題が発生したフェーズの歯車を停止する。
- 「再試行」「プロジェクトを変更」「セットアップを中止」を状況に応じて表示する。
- 内部スタックトレースは「技術的な詳細」に隠す。
- 自動で品質の低い代替処理へ切り替えない。

## 性能とコスト

- プロジェクト理解は入口ファイルを絞って一回の統合判断で行う。
- 基礎三体と初期専門家を作るために、各エージェントの準備ターンを実行しない。
- 専門家スレッドは役割契約を持った待機状態として作る。
- 同じレビュー、同じクリック負荷試験、同じ状態検証をフェーズごとに繰り返さない。
- 初回セットアップ完了時の確認は一回だけにする。
- 外部資産探索は必要領域の記録までとし、本格探索は実タスクで行う。

## 実装段階とユーザー確認

### 第1実装: 見た目の成立確認

- 専用セットアップ画面を独立コンポーネントとして追加する。
- 左側の処理詳細、背景のパイプオルガン、右側の六段歯車とフェーズ表示を作る。
- フェーズ3のレビュー用状態を表示する。
- 実データ契約へ接続できる型を先に用意する。
- 実エンジン、インストーラー連携、ファイル書き込みはまだ行わない。
- ユーザーが実画面を確認したところで停止する。

### 第2実装: 入力フロー

- プロジェクト確認
- 新規作成と既存選択
- 名前と説明の自動入力、編集
- 0件から3件の補完質問
- セットアップ開始確認

### 第3実装: セットアップエンジン

- `session.json`
- 六段階の状態機械
- 再開、取消、エラー処理
- 三体の基礎エージェント
- 完成マップ
- 初期専門家編成

### 第4実装: インストールと移行

- Codexからのプロジェクトルート引き継ぎ
- Windowsインストール後の自動起動
- 手動インストール時の分岐
- 旧五体構成から三体構成への移行
- 旧ウェブセットアップの廃止または互換読み取り

各実装段階の最初の画面をユーザーへ見せ、承認前に次の段階へ進まない。

## 第1実装の受け入れ条件

- 通常ホーム画面を変更せず、専用セットアップ画面を表示できる。
- 1440×900と1366×768で主要情報が切れない。
- 左側に現在処理と直近の完了、次の処理が読める。
- 中央背景に、ロゴではないパイプオルガンが存在する。
- 右側に六つのフェーズと六段の歯車が存在する。
- 完了、実行中、待機中の差が色だけに依存しない。
- 実行中の歯車だけが動く。
- reduced motionで動きを止めても状態が分かる。
- 技術的詳細を開閉できる。
- セットアップ中止操作を確認付きで実行できる。
- 画面はレビュー用の状態契約から描画され、固定HTMLに処理名を埋め込まない。
- ユーザーが実機で確認できる入口を用意する。

## 非目標

第1実装では次を行わない。

- セットアップエンジンの実行
- `.orquesta`への書き込み
- Codexスレッドの作成
- Windowsインストーラー連携
- 旧エージェントの移行
- パイプオルガンの音声再生
- 最終的なホーム遷移
- 全フェーズの完成品質までの作り込み

第1実装の目的は、ビジュアル表現がOrquesta Desktopとして成立するかを早期に判断することに限定する。
