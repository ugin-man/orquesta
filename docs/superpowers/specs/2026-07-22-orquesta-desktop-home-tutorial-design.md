# Orquesta Desktop Homeチュートリアル設計

## 目的

初めてOrquesta DesktopのHomeを開いたユーザーが、画面全体を覚えなくても、次の三つを理解できる状態にする。

- どこを見れば、現在の組織と作業状況が分かるか
- どこから、統括者へ指示を出せるか
- 自分の対応、過去の記録、設定をどこで確認するか

チュートリアルは操作練習ではなく、実際のHome画面を使った説明とする。チュートリアル中は説明対象以外を暗くし、Homeの操作は止める。ユーザーが使い始めるまでの時間を長くしないため、歓迎ページと完了ページは作らない。

## 採用する表示方式

画面全体へ半透明の暗幕を置き、説明対象だけを切り抜くスポットライト方式を採用する。説明カードには、現在位置、説明、戻る、次へ、スキップを表示する。

候補は次の三つを比較した。

| 方式 | 長所 | 問題 | 判断 |
| --- | --- | --- | --- |
| SVGマスクで対象を切り抜く | 実際のHomeを見せられる。重なったUIでも暗幕を安定して出せる | 対象位置の計測が必要 | 採用 |
| 対象以外のDOMを個別に暗くする | 実装量が少ない | z-indexやポップアップ追加で壊れやすい | 不採用 |
| 画像や専用スライドで説明する | 表示が安定する | 実際の位置を覚えにくく、Home変更で画像が古くなる | 不採用 |

チュートリアル中は、強調された対象も含めて通常操作を受け付けない。操作で画面状態が変わって説明対象が消えることを防ぐためである。Escapeキー、スキップ、次へ、戻るだけを受け付ける。

## ページ数

合計7ページとする。

7ページには、Homeを使うために必要な機能と、Homeから詳細画面へ移るための入口を含める。歓迎だけのページと完了だけのページは数に含めない。最後のページの「完了」でそのままHomeを使える状態に戻す。

6ページ以下にすると、Project操作、Luca、Team管理の説明を一枚へ詰め込むことになり、どこを見ればよいか分かりにくくなる。8ページ以上にすると、初回利用前の拘束時間が長くなりすぎる。各ページを読む時間は10秒前後、全体は60秒から90秒を目安にする。

## 説明順

順番は、Orquestaを理解するための中心機能、ユーザーが直接操作する機能、現在の確認、詳細画面への入口、補助機能の順とする。

### 1ページ目 中央のOrquestaマップ

重要度は最上位。Homeを開く理由そのものなので、最初に説明する。

強調する範囲は、中央マップ、ズーム操作、チーム管理ボタンを含むマップ領域全体とする。

表示する説明は次の内容とする。

> ここがOrquestaの組織図です。統括者、専門家、チーム、生産ラインを一つの地図として表示します。待機中のエージェントも含め、現在存在するエージェントはすべてここに残ります。エージェントやタスクを選ぶと、役割、状態、担当作業の詳細を確認できます。ドラッグで移動し、ホイールまたは上部のボタンで拡大できます。

このページでは、次の点を短く伝える。

- マップは作業履歴ではなく、現在の組織構造を表す
- 稼働中、待機中、停止中などの状態が見える
- エージェントやタスクを選ぶと詳細を開ける
- チーム管理は、組織と一時監査を管理する入口である

### 2ページ目 統括者への入力欄

重要度は高い。ユーザーがOrquestaを動かす主操作なので、マップの次に説明する。

強調する範囲は、画面下部の入力欄全体とする。

表示する説明は次の内容とする。

> Orquestaへ指示や質問を送る場所です。通常は統括者へ送り、統括者が必要な専門家へ作業を分けます。送信先の変更や画像の添付もここから行えます。送信できたことと、タスクが完了したことは別なので、進行状況はマップと「現在」で確認します。

このページでは、送信先、本文、添付、送信状態を一つの入力機能として説明する。細かなショートカットや対応ファイル形式は説明しない。

### 3ページ目 ユーザータスクと要対応

重要度は高い。質問や承認を見落とすと作業が止まるため、現在表示より先に説明する。

強調する範囲は、Home右側の要対応カードと、カード内のユーザータスク件数とする。

表示する説明は次の内容とする。

> ここには、あなたの対応を待っている項目だけが表示されます。質問、承認、確認、手作業が対象です。項目を開くとHome上で短い内容を確認でき、すべてを見たい場合は「ユーザータスク」へ移動できます。回答済みや完了済みの項目は、Homeへ残し続けません。

質問、承認、確認、手作業の違いを一行ずつ並べない。種類名を見せ、共通して「ユーザーが処理する項目」であることを優先する。

### 4ページ目 現在

重要度は中から高。ユーザータスクとは目的が違うため、別ページで説明する。

強調する範囲は、Home左側の「現在」カード全体とする。

表示する説明は次の内容とする。

> 今まさに動いている作業を短く確認する場所です。実行証拠がある作業を優先して表示し、未完了タスクのすべてや過去の履歴を並べる場所ではありません。作業を選ぶと詳細を確認でき、全体は「記録」から確認できます。

「現在に表示されないタスクが存在する」ことを明示する。Homeの件数と記録内の未完了件数が違っても、直ちにバグとは限らないと分かる説明にする。

### 5ページ目 下部の画面切り替え

重要度は中。Homeだけで全情報を表示しないという設計を理解してもらうページである。

強調する範囲は、下部のHome、ユーザータスク、記録、設定の四つをまとめたDockとする。

表示する説明は次の内容とする。

> Homeは、今の全体像を見る画面です。詳しい情報は下の四つから切り替えます。ユーザータスクはあなたの対応、記録はタスク・エラー・会話・判断・タイムライン、設定は表示・Codex接続・起動・診断を扱います。

カード内には次の短い一覧も表示する。

- Home：現在の全体像
- ユーザータスク：自分が対応する項目
- 記録：タスク、エラー、会話、判断、タイムライン
- 設定：表示、接続、起動、診断

チュートリアル中に実際のタブへ移動はしない。Home上でDock全体を見せて説明を終える。

### 6ページ目 Project操作と状態

重要度は中から低。日常的に毎回触る機能ではないが、複数Projectを使う場合に必要なので説明する。

強調する範囲は、左上のProject切り替えと、右上のProject状況を同時に切り抜く。スポットライトが複数の穴を持つ唯一のページとする。

表示する説明は次の内容とする。

> 左上からProjectを切り替えたり、現在のProjectフォルダを開いたりできます。右上では、Project名、エージェント数、接続状態を確認できます。詳細な診断は設定にまとめています。

上部中央のRepository状態表示は、右上のProject状況の補足として一文だけ説明する。単独ページにはしない。

### 7ページ目 Luca

重要度は補助機能として低めなので最後に説明する。ただし、ユーザーが現在の状況を理解できない場合の出口として、存在は必ず案内する。

強調する範囲は、HomeのLucaボタンとする。

表示する説明は次の内容とする。

> 何が起きているか分からないときは、Lucaへ聞けます。Home全体、タスク、エラー、監査について、現在のOrquesta記録をもとに説明します。Lucaは説明役であり、質問しただけでタスクや組織を変更することはありません。

最後の操作は「次へ」ではなく「完了」とする。完了後は暗幕を外し、Homeの状態を変えずに操作可能へ戻す。

## 説明しないもの

次の内容は初回チュートリアルへ入れない。

- すべてのステータス色と内部的な証拠分類
- タスク、エラー、会話、判断、タイムラインの細かなフィルター
- 外部比較と敵対監査の実行方法
- Project Routeと運用状況の詳細
- Codex App Serverの技術情報
- 初回セットアップの六段階

これらは、各詳細画面の空状態説明、ヘルプ、または将来の個別チュートリアルで扱う。Homeチュートリアルへ混ぜない。

## 説明カード

説明カードには次だけを置く。

- ページ番号。例 `1 / 7`
- 短い見出し
- 説明文
- 戻る
- 次へ、最終ページでは完了
- スキップ

一ページの本文は日本語で180文字前後を上限とする。マップとDockだけは補助の短い箇条書きを許可する。説明カードは対象を隠さない側へ自動配置し、上下左右の空きが不足する場合は画面内へ収まる位置まで移動する。

対象の外側は黒ではなく、Homeの温かい色を残す薄い墨色で暗くする。強調枠は白く発光させず、Orquestaで使っている緑と薄い白の二重線にする。常時点滅は使わず、ページが変わった瞬間だけ150ミリ秒から220ミリ秒で位置を移す。OSの「動きを減らす」が有効なら即時切り替えにする。

## 開始条件と再実行

正式な動作は次の通りとする。

1. 初回セットアップが完了する
2. Homeの主要部品が描画される
3. 現在のチュートリアルversionを完了またはスキップした記録がない
4. 1ページ目から自動開始する

チュートリアルの表示履歴はProjectの`.orquesta`へ保存しない。これは組織状態ではなく、このPC上の利用者向け表示設定だからである。Desktopのapp-owned設定へ、チュートリアルversionと結果を保存する。

保存する状態は次の三つで足りる。

```ts
type HomeTutorialPreference = {
  version: 1;
  outcome: 'completed' | 'skipped';
  updatedAt: string;
};
```

途中でアプリを閉じた場合は、完了またはスキップとして保存しない。次回のHome表示で1ページ目から再開する。細かな途中位置は保存しない。

初回セットアップ画面が完成する前は、自動開始を接続しない。設定の「表示」に「ホーム画面のチュートリアルを開始」を置き、そこから手動確認できるようにする。初回セットアップ側が完成した後、完了イベントから同じ開始処理を一度だけ呼ぶ。

## 画面構造

チュートリアル機能はHomeの各部品へ説明処理を埋め込まない。次の三つに分ける。

- `HomeTutorialController`：開始、戻る、次へ、完了、スキップ、保存状態を管理する
- `HomeTutorialOverlay`：暗幕、スポットライト、説明カード、キーボード操作を描画する
- `TutorialTargetRegistry`：マップ、入力欄、要対応、現在、Dock、Project操作、LucaのDOM位置を提供する

対象はCSS class名を検索せず、チュートリアル用の安定したtarget IDで登録する。Homeの見た目を変更しても、説明対象の意味が同じならチュートリアルを壊さないためである。

一つのページは次のデータで定義する。

```ts
type HomeTutorialStep = {
  id: 'map' | 'composer' | 'user-tasks' | 'now' | 'dock' | 'project' | 'luca';
  priority: 1 | 2 | 3 | 4 | 5 | 6 | 7;
  targetIds: string[];
  title: { ja: string; en: string };
  body: { ja: string; en: string };
};
```

Targetが一時的に存在しない場合は、画面中央へ説明カードだけを出さず、そのページを自動で飛ばす。複数のTargetが必要な6ページ目は、一方だけ存在する場合、存在する方だけを強調して説明を続ける。全Targetがない場合だけ飛ばす。

## 操作とアクセシビリティ

- チュートリアル開始時に説明カードへフォーカスを移す
- Tabキーの移動を説明カード内へ閉じる
- Escapeキーはスキップ確認を出さず、そのversionをスキップ扱いにして閉じる
- 左右矢印キーで戻る、次へを操作できる
- 背景のボタンとマップはマウス、キーボードとも操作不能にする
- スクリーンリーダーには説明カードをdialogとして伝え、ページ変更時に見出しと本文を読み上げる
- window resize、表示倍率変更、Home内のカード拡大が起きたら対象位置を再計測する

## 初回セットアップ担当との境界

Homeチュートリアル側は、初回セットアップの画面、SetupEngine、六段階表示を変更しない。自動開始に必要なのは「セットアップが完了し、Homeを表示できる」という一つの通知だけである。

先に実装する範囲は次とする。

- 7ページの定義
- Home上のTarget登録
- スポットライトと説明カード
- 設定の「表示」から手動開始
- 完了、スキップ、再実行

初回セットアップ担当の変更を取り込んだ後に、完了通知からの自動開始だけを接続する。この順番なら、両方が同時に`DesktopRendererApp`の初回分岐を大きく変更する競合を避けられる。

## 確認項目

実装後は、同じ動きを何度も検証せず、次の境界だけを確認する。

- 7ページが決めた順番で進み、戻れる
- 各ページで正しいHome領域だけが強調される
- チュートリアル中に背景をクリックしてもHomeが変化しない
- スキップ後は自動再表示されず、設定から再実行できる
- 完了後はHomeの状態が保持される
- Targetが一つ欠けてもチュートリアル全体が停止しない
- 日本語と英語でカードが画面外へ出ない
- 1440×900と1280×720のDesktopウィンドウで対象とカードが重ならない
- 「動きを減らす」で移動アニメーションが止まる

最終確認はElectronのDesktop画面で行う。ブラウザ表示はレイアウト調整の中間確認に限り、完成判定には使わない。
