# Orquesta Desktop 起動画面設計

## 目的

現在の透過ウィンドウを使ったスプラッシュ表示をやめる。通常のOrquesta Desktopウィンドウが開いた直後、その全面にロゴを表示し、読み込み完了後にロゴが上へ消えてホーム画面が現れる構成にする。

起動画面は演出だけではなく、リポジトリ情報を読み込む間の待機画面も兼ねる。ただし読み込みが速くてもロゴを一瞬で消さず、ブランドを認識できる時間を確保する。

## 採用する見た目

ユーザーが提示した「写真1」のロゴをそのまま使用する。黒を基調にした細めの文字と広い余白が、現在の白黒と温かい紙色を使った画面に合うため、写真2ではなく写真1を採用する。

実装時は元画像を `apps/orquesta-desktop/public/brand/orquesta-startup.jpg` にコピーする。CSSで描き直したり、別のロゴへ置き換えたりしない。

起動中の背景は画像の白とつながる白にする。ロゴ画像は中央に置き、ウィンドウの大きさに合わせて縮小する。通常の1440×900では画像全体を約420px四方とし、小さい画面では縦横の52%を上限にする。

## 採用する方式

`index.html` に起動カーテンを置き、その背後で通常のReact画面を読み込む方式にする。

Reactだけで起動画面を作ると、JavaScriptが起動するまで白画面が見える可能性がある。別の透過BrowserWindowを使い続けると、二つのウィンドウの表示順、終了時刻、フォーカスを同期する必要が残る。HTMLに最小限の起動カーテンを置けば、メインウィンドウの最初の描画からロゴを表示でき、読み込み中も同じ一つのウィンドウで完結する。

この変更では次の構成にする。

- 透過スプラッシュ用BrowserWindowを廃止する
- 通常のメインウィンドウを `ready-to-show` で表示する
- `index.html` の起動カーテンを `#root` より前面に固定する
- Reactはカーテンの背後でリポジトリ、ホーム画面、エラー画面を準備する
- 準備完了と最低表示時間の両方を満たしたら、レンダラー側からカーテンを終了させる

## 表示の流れ

1. メインウィンドウが開き、白い全面カーテンと写真1のロゴが中央に表示される。
2. 背後で現在のプロジェクトとホーム画面を読み込む。
3. ロゴは読み込みが早く終わっても、最初の描画から1400msは中央に静止する。
4. 読み込み完了後、ロゴを300msかけて約24px上へ動かしながら透明にする。同時に白いカーテンも薄くなり、その下の画面が現れる。
5. アニメーション終了後、起動カーテンをDOMから削除し、通常操作を有効にする。

読み込みに1400ms以上かかった場合は、準備完了後すぐに終了アニメーションを始める。6秒経っても準備完了の通知が来ない場合は、カーテンを終了して既存の読み込み中画面またはエラー画面を見せる。ロゴのまま永久に止まる状態は作らない。

ホーム画面そのものは動かさない。ロゴと白いカーテンだけが消えることで画面を見せる。画面全体まで大きく移動させると、起動のたびに目が疲れ、現在の静かなデザインとも合わないためである。

## 起動完了の判定

通常時は、リポジトリのスナップショットを取得できた時点を準備完了とする。取得に失敗した場合も、既存のエラー表示を準備できた時点で完了扱いにする。成功時だけ起動画面を閉じる設計にはしない。

現在の `rendererReady` 通知と別ウィンドウ用のreadiness gateは役目が重複するため、起動カーテンへの移行時に整理する。最低表示時間、準備完了、6秒の上限はレンダラー側の小さな起動制御にまとめる。

プロジェクト切り替え、ホームへの移動、ウィンドウの再フォーカスでは、この演出を再生しない。アプリを新しく起動したときだけ表示する。

## 操作とアクセシビリティ

起動中はカーテンがクリックやキーボード操作を受け止め、背後の画面を誤操作できないようにする。カーテン自体は読み上げ対象にせず、フォーカス可能な要素も置かない。

OSの「アニメーションを減らす」が有効な場合は、上方向への移動を行わず、短いフェードだけにする。最低1400msのロゴ表示は維持する。

## 検証範囲

実装時は次だけを確認する。

- 起動直後に旧透過スプラッシュではなく、通常ウィンドウ内に写真1が表示される
- 読み込みが速くても1400msより前に消えない
- 読み込みが遅い場合は準備完了まで残り、その後300msで消える
- 読み込み失敗時と6秒の上限でも操作可能な画面へ進む
- 小さいウィンドウでロゴがはみ出さない
- reduced motionでは移動せずフェードだけになる
- 起動後にカーテンが残らず、ホーム画面を通常どおり操作できる

既存のE2Eと画面検証では、起動演出を即時終了できるテスト用指定を用意する。毎回1.7秒待つテストにはしない。

## 今回変更しないもの

この作業ではホーム画面の配置、オルケスタマップ、ユーザータスク、記録、設定、プロジェクト切り替え、Codexとの通信を変更しない。現在作業中のホーム画面の微調整ともコミットを分ける。

過去の設計書にある「透明背景の別スプラッシュ」と「650〜900msで終了する」という記述は、起動画面に関する部分だけ本設計で置き換える。
