## はじめに

Codexを使って開発していると、ある地点から「一人の万能AIに全部任せるのは厳しい」と感じるようになりました。

小さな修正や、仕様が明確な実装なら問題ありません。けれど、ゲーム制作やUI改善のように、設計、実装、ビジュアル、世界観、QA、ユーザーの感覚が混ざる作業では、ひとつのチャットに全てを詰め込むほど苦しくなります。

そこで作っているのが **Orquesta** です。

Orquestaは、Codexを「一人の作業者」として使うのではなく、長期稼働する専門AIチームとして運用するためのCodex skillです。

GitHubはこちらです。

https://github.com/ugin-man/orquesta

## 何に困っていたのか

AIコーディングは速いです。

ただ、速いぶんだけ、プロジェクトが大きくなると次の問題が目立ちます。

- 一つのチャットに読み込ませる文脈が増えすぎる
- UI、実装、物語、QAの話が同じ場所に混ざる
- ユーザーの細かい感覚が、要約の過程で落ちる
- サブエージェントが使えない、または使いづらいワークフローがある
- いま誰が何をしているのか分からなくなる
- 承認待ちや環境エラーで、作業が静かに止まる

特にゲーム制作では、これはかなり深刻です。

ゲームはコードだけではなく、ビジュアル、操作感、世界観、テンポ、UIの密度、プレイヤーに与える印象の組み合わせです。ユーザー側の頭の中にある「こうしたい」という感覚を、最初から全部言語化するのはほぼ無理です。

だから、AIが勝手に完成まで走り切るよりも、ユーザーとAIチームが何度もすり合わせながら進む仕組みが必要だと考えました。

## なぜサブエージェントではないのか

サブエージェントは便利です。

ただ、Orquestaでは最初から「通常のCodexスレッドを、長期で働く専門家として扱う」設計にしています。

理由は主に3つです。

1つ目は、長期的な文脈を持たせたいからです。

例えばビジュアル担当AIには、ビジュアルの規律、過去の判断、ユーザーの好みだけを長く持っていてほしい。実装担当AIには、コード構造やテスト方針を持っていてほしい。世界観担当AIには、世界設定やストーリーの前提を持っていてほしい。

全部のAIが全部の文書を読む必要はありません。

2つ目は、ユーザーが専門AIと直接話せるようにしたいからです。

ビジュアルの微妙な違和感を、毎回統括AIに説明して要約してもらうより、ビジュアル担当AIと直接話した方がいい場面があります。Orquestaでは、専門AIとの直接会話を正式な機能として扱い、その内容をあとで統括側の状態に同期する前提にしています。

3つ目は、サブエージェントを使えないタスクやスキルがあるからです。

画像生成系や、特定のプラグイン、外部ツールを含む作業では、サブエージェントではうまく動かないケースがあります。だから最初から、通常スレッドを管理する仕組みに寄せています。

## Orquestaの基本構造

Orquestaをプロジェクトに入れると、まず基礎チームを作ります。

- `orchestrator`: 統括。タスクの分配、状態管理、報告の受理を行う
- `user-liaison`: ユーザー側の作業、承認、質問を整理する
- `vision-curator`: ユーザーの回答や好みを、採用前の提案に整理する
- `error-concierge`: エラーや権限問題を整理し、ユーザー側で解決できることを提示する
- `orquesta-admin`: Orquesta自体の設定やダッシュボード導線を担当する

その後、プロジェクトに応じて実装、ビジュアル、世界観、QAなどの専門AIを作ります。

重要なのは、Orquestaが「完全自律で勝手に完成させるシステム」ではないことです。

ユーザーは単なる依頼者ではなく、プロジェクトの創造的な起点です。Orquestaは、そのユーザーと一緒に作るための運用システムです。

## ダッシュボード

Orquestaにはローカルダッシュボードがあります。

ここでは、次のようなものを確認できます。

- どんなAIチームが作られているか
- どのAIが動いているか
- どのタスクが進行中か
- どこで承認待ちになっているか
- ユーザーが答えるべき質問や作業
- 完成までの大きな地図

ダッシュボードは、単なる見た目ではありません。

マルチエージェント運用で一番怖いのは、「誰かが何かをしている気がするけど、実際には止まっている」状態です。Orquestaでは、状態ファイルとダッシュボードを使って、チームの現在地を見えるようにしようとしています。

## 質問とビジョン共有

Orquestaでは、プロジェクト開始時にユーザーへ質問します。

これは、AIに命令を確定させるためだけではありません。

ユーザー自身がまだ言語化できていない感覚を掘り起こすためでもあります。

ただし、ユーザーの回答をそのまま絶対条件として採用するのは危険です。回答は「議論の種」として扱い、`vision-curator` が強いシグナル、候補ルール、確認が必要な点に整理します。

その後、統括AIが採用するか、修正するか、保留するかを判断します。

このクッションを入れることで、「質問に答えたら、そのまま変な方向に実装された」という事故を減らしたいと考えています。

## beta.2でできること

現在のOrquestaは `0.1.0-beta.2` です。

できるようになっていることは、だいたい次の通りです。

- GitHubからskillとしてインストールできる
- 初回セットアップで基礎チームを作る
- ローカルダッシュボードを起動する
- ポート衝突時に空きポートへ逃がす
- ダッシュボードでチーム、タスク、ユーザー作業を表示する
- ユーザーへの質問と回答を扱う
- 専門AIへのハンドオフ文面を作る
- 専門AIの報告を受理または差し戻す
- 承認待ちが静かに詰まらないよう、ユーザータスクとして表示する
- delegation gateにより、統括AIが専門作業を勝手に吸収しないよう検査する

一方で、まだ弱いところもあります。

- 初回セットアップはもっと簡単にしたい
- 実プロジェクトでの検証数が少ない
- Codex環境によって使えるスレッド操作ツールに差がある
- ダッシュボードや状態ファイルの仕様はまだ変わる可能性が高い
- 英語圏ユーザー向けの説明やデモがまだ足りない

## 使ってみる

インストール先はこのGitHubリポジトリです。

https://github.com/ugin-man/orquesta/tree/main/orquesta

手元で試す場合は、新しいCodexスレッドを開いて、対象プロジェクトで次のように依頼します。

```text
Use the orquesta skill to set up this project.
```

その後、作りたいものを説明します。

```text
I want to make a browser game. Please use Orquesta to set up the project team and dashboard.
```

詳しい入口はリポジトリの `START_HERE.md` にまとめています。

## どういう人に試してほしいか

今いちばん試してほしいのは、次のような人です。

- CodexやClaude Codeなどで、AI開発ワークフローをかなり使っている
- 一つのAIに全部任せる運用に限界を感じている
- ゲーム、UI、創作系ツールなど、感性のすり合わせが多いものを作っている
- マルチエージェント運用に興味がある
- 「完全自律」より「ユーザーとAIチームが一緒に作る」方向に興味がある

まだ完成品ではありません。

でも、同じような痛みを持っている人がいたら、一緒に壊して、一緒に直してほしいです。

## おわりに

Orquestaは、AIに会社ごっこをさせたいわけではありません。

やりたいのは、ユーザーの頭の中にある曖昧なビジョンを、長く一緒に働くAIチームと少しずつ形にしていくことです。

一人のAIに全部を背負わせるのではなく、それぞれの専門家が狭い文脈を持ち、ユーザーが必要な相手と直接話し、統括が状態を同期する。

そういうAI開発の形を試しています。

興味がある人は、ぜひGitHubを見てください。

https://github.com/ugin-man/orquesta
