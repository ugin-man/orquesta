ここでいうCodexは、昔のOpenAI Codexモデルではなく、今のCodex環境で動く開発エージェントのことです。
ローカルファイルを読み書きしたり、ブラウザやGitHubやスキルを使ったりしながら開発を進める相手、くらいの意味で読んでください。

## ひとつのCodexスレッドが大きくなりすぎる

Codexで開発していると、最初はひとつのスレッドでかなり進められます。
小さな修正、明確なバグ修正、単発の実装なら、それで十分です。

でも、プロジェクトが少し大きくなると、だんだん別の問題が出てきます。

- ひとつのスレッドに読み込ませる文脈が増えすぎる
- 実装、UI、QA、設計、ユーザー対応の話が同じ場所に混ざる
- 前に決めた役割や判断が、要約やコンテキスト圧縮でぼやける
- 統括役のAIが、いつの間にか専門作業を直接やり始める
- 誰が何をしていて、どこで止まっているのか見えにくくなる

特にゲーム開発やUIの多いソフトウェアでは、この問題が強く出ます。
コードだけではなく、画面の印象、操作感、世界観、QA、ユーザーの細かい感覚が絡むからです。

そこで作っているのが Orquesta です。

https://github.com/ugin-man/orquesta

Orquestaは、Codexを「ひとりの巨大な作業者」として使うのではなく、長期専門AIチームとして運用するためのCodex skillです。

## Orquestaがやりたいこと

Orquestaの中心には `orchestrator` がいます。
ただし、orchestratorは万能実装者ではありません。

orchestratorの役割は、作業を分類し、適切な専門スレッドへ渡し、報告を受け取り、状態を同期することです。
実装、ダッシュボードUX、文書、QA、プロトコル、構想整理、障害整理などは、それぞれの専門AIが担当します。

Orquestaでは、主に次のような役割を扱います。

- `orchestrator`: ルーティング、状態管理、報告の受理
- `user-liaison`: ユーザー側の承認、質問、作業待ちの整理
- `vision-curator`: ユーザーの回答や好みを、採用前の方向性候補として整理
- `error-concierge`: 環境エラーや権限問題を、修理カードのように整理
- `orquesta-admin`: Orquesta自体の設定、初期セットアップ、ダッシュボード導線
- production specialists: 実装、UI/UX、QA、文書、プロトコルなど、プロジェクトごとの専門係

重要なのは、Orquestaが「AIに全部自律実行させる仕組み」ではないことです。
ユーザーは依頼者というより、創造的な起点です。
Orquestaは、ユーザーと専門AIチームが一緒に作るための運用レイヤーです。

## なぜサブエージェントだけではなく、長期スレッドなのか

サブエージェントは便利です。
調査、ログ確認、独立した小さな作業にはかなり向いています。

ただ、Orquestaでは通常のCodexスレッドを長期専門AIとして扱います。
これは、サブエージェントを否定したいからではありません。
サブエージェントでは扱いにくい規模の仕事や、プラグインを使う必要がある仕事を、継続して動かしたかったからです。

まず、専門ごとの文脈を長く持たせたいという理由があります。

文書担当には、README、リリースノート、公開説明、インストール導線の文脈を持っていてほしい。
ダッシュボード担当には、画面構成、操作感、ユーザーが迷う場所を持っていてほしい。
実装担当には、コード構造とテスト方針を持っていてほしい。

全部のAIが全部の文書を読む必要はありません。
強い文脈を、それぞれの専門に閉じ込めたい。

それから、ユーザーが専門AIと直接話せるようにしたいという理由もあります。

UIの違和感や文章の温度感のような話は、統括AIを経由して要約するより、担当AIと直接話した方が早い場面があります。
Orquestaでは、その直接会話を正式な機能として扱い、あとで `.orquesta/` の状態やレポートへ同期します。

もうひとつ大きいのが、プラグインやローカル環境との相性です。

画像生成、ブラウザ操作、GitHub、文書生成、ローカルUI確認のような作業は、サブエージェントだとうまく起動できなかったり、必要なプラグインを使えなかったりすることがあります。
一回きりの補助作業ならそれでもよいのですが、Orquestaでやりたいのは、専門AIが役割を持ち、必要な道具を使いながら、何度も仕事を受けて返す運用です。

だから、通常のCodexスレッドを専門AIとして立てています。
文脈を持てること、ユーザーと直接話せること、プラグインやローカル確認を使えること。
これらがそろって、ようやく「専門AIチーム」として動かせる感覚があります。

## いま動いているもの

現在のOrquestaは `0.1.0-beta.2` です。
完成品ではありませんが、次のものは動いています。

### file-backedなOrquesta状態

プロジェクト内に `.orquesta/` を作り、エージェント、セッション、タスク、ユーザー側作業、障害、構想整理、レポートなどをファイルとして保存します。

チャット履歴だけに頼らないので、コンテキスト圧縮が起きても運用ルールを継続しやすくなります。

### mission-control型のローカルダッシュボード

ローカルダッシュボードでは、チーム構造、タスク、ユーザー側作業、プロジェクトルート、レポート確認、承認待ちなどを見られます。

今の方向性は、単なるステータスページではなく、mission-control / Command Board 風の画面です。
glass風の視覚表現、専門AIのマップ、ユーザーアクションのレール、プロジェクト進行の表示を持っています。

ただし、ダッシュボードがあるだけではマルチエージェント運用の証明にはなりません。
実際に専門AIへhandoffされ、報告が返り、orchestratorが受理または差し戻しした証拠が必要です。

### delegation gate

Orquestaでは、専門領域の作業をorchestratorがこっそり吸収しないように、delegation gateを置いています。

専門AIが担当すべきタスクなら、状態には次のような証拠が必要です。

- どの専門AIへ渡したか
- handoffが送られたか
- 専門AIの報告や成果物があるか
- orchestratorが受理、保留、差し戻しの判断をしたか

これは、READMEで「長期専門AIチーム」と言っているのに、実際は統括スレッドが全部やっている、という状態を避けるためのものです。

### user-side tasks

AI側だけで進められないこともあります。
たとえば、ユーザーの承認、方針確認、ローカル環境での操作、権限の許可などです。

Orquestaでは、そうしたものをチャットの中に埋もれさせず、ユーザー側タスクとして扱います。
`user-liaison` がその窓口になります。

### vision / failure / user liaison layers

Orquestaには、プロジェクトを進めるための補助レイヤーがあります。

- vision layer: ユーザーの回答や好みを、採用前の方向性候補として整理する
- failure concierge: 環境エラーや繰り返す失敗を、修理カードのように整理する
- user liaison: ユーザーが答えるべきこと、承認すべきこと、待っていることを整理する

これらは常時監視エージェントではありません。
必要な条件が出たときに起こす、event-drivenな役割です。

### trigger audit と question-candidate visibility

最近入れた仕組みとして、trigger auditがあります。

これは、`vision-curator` などのfoundation roleを常時 watcher にするのではなく、「いま起こすべき条件があるか」を監査結果として見えるようにするものです。

たとえば専門AIのレポートに、ユーザーへ聞いた方がよい質問候補が含まれている場合、それをすぐ生の質問として表示するのではなく、question candidateとして扱います。

trigger auditは、pending question candidates が一定数ある、高優先度である、受理前に確認すべきである、といった条件を見えるようにします。

でも、trigger audit自体は `vision-curator` を勝手に起動しません。
raw candidateを質問として昇格もしません。
orchestratorが、起こすか、保留するか、別のルートにするかを判断します。

## まだbetaです

ここは強調しておきたいです。
Orquestaはまだbeta品質です。

できることは増えていますが、安定版でも、完成品でも、本番運用に耐えると断言できるものでもありません。

今の弱いところは、たとえば次のあたりです。

- もっと多くのクリーンプロジェクトで試す必要がある
- Codex環境によって、使えるスレッド操作やツールが違う
- ダッシュボードや状態ファイルの仕様はまだ変わる可能性が高い
- 公開ドキュメントと実運用がズレないよう、継続して確認が必要
- 日本語・英語の説明、導入記事、デモがまだ足りない

逆に言うと、今は「一緒に壊して、一緒に直す」人がとてもほしい段階です。

## どう試すか

インストール対象はこのリポジトリの `orquesta/` です。

https://github.com/ugin-man/orquesta/tree/main/orquesta

手元で試す場合は、新しいCodexスレッドを対象プロジェクトで開き、次のように依頼します。

```text
Use the orquesta skill to set up this project.
```

日本語なら、たとえばこうです。

```text
orquestaスキルでこのプロジェクトを初期セットアップしてください。
```

入口の説明は `START_HERE.md` にまとめています。

https://github.com/ugin-man/orquesta/blob/main/START_HERE.md

## ほしいフィードバック

今ほしいのは、きれいな称賛よりも、実際に使ったときの引っかかりです。

- インストールできたか
- 最初の説明で何をすればいいか分かったか
- ダッシュボードに実際のチーム状態が見えたか
- 専門AIへの委任が本当に起きたか
- ユーザー側タスクや承認待ちが分かりやすかったか
- どこが重い、怖い、分かりにくいと感じたか

特に、Codexで大きめの個人開発、ゲーム制作、AI多めのツール、創作系プロジェクトをやっている人に試してほしいです。

## おわりに

Orquestaは、AIに会社ごっこをさせたいわけではありません。

やりたいのは、ひとつの巨大なAIチャットに全部を背負わせるのではなく、強い文脈を持った専門AIたちと、ユーザーが一緒に作れる状態を作ることです。

まだ荒いです。
でも、「AIに全部投げる」でも「人間が全部管理する」でもない、中間の作り方には可能性があると思っています。

もし同じ痛みを感じていたら、ぜひ触って、壊して、意見をください。
