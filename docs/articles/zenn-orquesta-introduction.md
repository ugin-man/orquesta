個人開発で、Codex上で動くマルチエージェントシステムを作っています。
名前は Orquesta です。

https://github.com/ugin-man/orquesta

ここでいうCodexは、昔のOpenAI Codexモデルではなく、現在のCodex環境で動く開発エージェントのことです。
ローカルファイルを読み書きしたり、ブラウザやGitHubやスキルを使ったりしながら開発を進める相手、くらいの意味で読んでください。

Orquestaは、Codexを「ひとつの大きな作業スレッド」として使うのではなく、複数の長期専門AIスレッドを持つチームとして運用するためのCodex skillです。

ただし、完全自律型のマルチエージェントを目指しているわけではありません。
ユーザーを間に置き、質問、承認、方針確認、作業状況の確認を含めて開発を進めるための仕組みです。

この記事では、実際に何を作っているのか、普通のマルチエージェント構成と何が違うのか、いま何が動いているのかを書きます。

## なぜ作ったのか

Codexで個人開発をしていると、最初はひとつのスレッドだけでかなり進められます。
小さな修正、明確なバグ修正、単発の実装なら、それで十分です。

でも、プロジェクトが大きくなると、次のような問題が出てきます。

- ひとつのスレッドに文脈が溜まりすぎる
- 実装、UI、QA、設計、文書、ユーザー対応が同じ場所に混ざる
- コンテキスト圧縮で、以前の判断や役割分担がぼやける
- 統括役のAIが、いつの間にか専門作業まで直接やり始める
- 誰が何をしていて、どこで止まっているのか見えにくい

特にUIやゲーム制作のように、コード以外の判断が多い開発では、この問題が強く出ます。
画面の違和感、操作感、文章の温度感、世界観、承認待ち、ローカル環境の問題などが、同じスレッドに積み重なっていくからです。

そこで、Codexの中に役割分担を持ったチームを作れないかと考えました。

## 作ったもの

Orquestaは、プロジェクト内に `.orquesta/` という状態ディレクトリを作り、AIチームの状態をファイルとして管理します。

扱う情報は、たとえば次のようなものです。

- エージェント一覧
- 専門スレッドのセッション情報
- タスクと委任状況
- ユーザー側の承認待ちや質問
- 障害や環境エラー
- 専門AIからのレポート
- ダッシュボード表示用の状態

中心には `orchestrator` がいます。
ただし、orchestratorは万能実装者ではありません。

orchestratorの役割は、作業を分類し、適切な専門スレッドに渡し、報告を受け取り、状態を同期することです。
実装、UI/UX、QA、文書、プロトコル、構想整理、障害整理などは、それぞれの専門AIが担当します。

たとえば、Orquestaでは次のような役割を扱います。

- `orchestrator`: ルーティング、状態管理、報告の受理
- `user-liaison`: ユーザー側の承認、質問、作業待ちの整理
- `vision-curator`: ユーザーの回答や好みを、採用前の方向性候補として整理
- `error-concierge`: 環境エラーや権限問題を、修理カードのように整理
- `orquesta-admin`: Orquesta自体の設定、初期セットアップ、ダッシュボード導線
- production specialists: 実装、UI/UX、QA、文書、プロトコルなど、プロジェクトごとの専門係

## 何が普通のマルチエージェントと違うのか

Orquestaの特徴は、大きく分けると4つあります。

### Codexの中で動く

Orquestaは、外部の独立したエージェント基盤として作っているものではありません。
Codexのスレッド、ローカルファイル、スキル、プラグイン、GitHub、ブラウザ確認などを使いながら動かすことを前提にしています。

つまり、普段Codexで開発している作業環境の中に、チーム運用のレイヤーを足すイメージです。

### サブエージェントだけに頼らない

Codexのサブエージェントは便利です。
調査、ログ確認、独立した小さな作業には向いています。

ただ、Orquestaでは通常のCodexスレッドを長期専門AIとして扱います。

理由は、サブエージェントでは扱いにくい規模の仕事があるからです。
長く文脈を持たせたい作業、ユーザーと直接話した方がよい作業、画像、ブラウザ、GitHub、文書生成、ローカルUI確認などのプラグインや環境が必要な作業では、長期スレッドの方が扱いやすい場面があります。

たとえば、文書担当にはREADMEや導入記事の文脈を持っていてほしい。
ダッシュボード担当には、画面構成やユーザーが迷った場所の文脈を持っていてほしい。
実装担当には、コード構造とテスト方針を持っていてほしい。

全部のAIが全部の文書を読むのではなく、専門ごとに必要な文脈を持たせるために、長期専門スレッドを使っています。

### 完全自律を目指さない

Orquestaは、AIだけで最後まで自動開発する仕組みではありません。

むしろ、ユーザーを開発ワークフローの中に明示的に入れます。
承認が必要なこと、方針確認が必要なこと、質問に答えた方がよいこと、ローカル環境でユーザーが操作した方が早いことを、ユーザー側タスクとして扱います。

これは単に「人間が邪魔をする」という話ではありません。
商用レベルの品質を目指すなら、ユーザーの違和感、優先順位、好み、判断、方向転換が必要になります。

Orquestaでは、専門AIが質問候補を出し、`vision-curator` が整理し、orchestratorが受け取り、必要なものをユーザー側に出す、という流れを想定しています。
ユーザーが答えた内容は、そのまま命令として採用するのではなく、方向性候補や判断材料として整理します。

AIが勝手に作るのではなく、ユーザーが開発状況を理解し、判断し、必要な質問に答えながら、品質を上げていくための仕組みです。

### 運用の証拠をファイルに残す

Orquestaでは、チャット上の雰囲気だけで「委任したことにする」のを避けます。

専門AIが担当すべき作業なら、状態ファイルに次のような証拠が必要です。

- どの専門AIへ渡したか
- handoffが送られたか
- 専門AIの報告や成果物があるか
- orchestratorが受理、保留、差し戻しの判断をしたか

この仕組みを `delegation gate` と呼んでいます。

これは、自分自身がOrquestaを作っている最中に、orchestratorが専門AIを使わずに作業してしまう問題があったために入れました。
マルチエージェントを名乗るなら、実際に委任された証拠が必要だと考えています。

## 実際のワークフロー

Orquestaで想定している流れは、だいたい次のようなものです。

1. ユーザーがCodexに作業を依頼する
2. orchestratorが作業を分類する
3. 必要なら専門AIにhandoffする
4. 専門AIが作業し、レポートや成果物を返す
5. orchestratorが受理、保留、差し戻しを判断する
6. ユーザー確認が必要なものは `user-liaison` がユーザー側タスクとして整理する
7. 質問候補や障害は、必要に応じて `vision-curator` や `error-concierge` が整理する
8. ダッシュボードで、組織図、タスク、承認待ち、質問、障害、レポートを確認する

重要なのは、ユーザーが状況を見失わないことです。

AIが何をしているのか、どこで詰まっているのか、どの判断をユーザーに求めているのかが見えないと、マルチエージェントはすぐにブラックボックスになります。
Orquestaでは、その部分をローカルダッシュボードとファイル状態で見えるようにしようとしています。

## いま動いているもの

現在のOrquestaは `0.1.0-beta.2` です。
完成品ではありませんが、次のものは動いています。

### file-backedなOrquesta状態

プロジェクト内に `.orquesta/` を作り、エージェント、セッション、タスク、ユーザー側作業、障害、構想整理、レポートなどを保存します。

チャット履歴だけに頼らないので、コンテキスト圧縮が起きても運用ルールを継続しやすくなります。

### ローカルダッシュボード

ローカルダッシュボードでは、チーム構造、タスク、ユーザー側作業、プロジェクトルート、レポート確認、承認待ちなどを見られます。

今の方向性は、単なるステータスページではなく、mission-control / Command Board 風の画面です。
専門AIのマップ、ユーザーアクション、プロジェクト進行、委任状況をまとめて確認できるようにしています。

ただし、ダッシュボードがあるだけではマルチエージェント運用の証明にはなりません。
実際に専門AIへhandoffされ、報告が返り、orchestratorが受理または差し戻しした証拠が必要です。

### delegation gate

専門AIが担当すべきタスクをorchestratorが吸収しないように、delegation gateを置いています。

専門AIへのhandoff、専門AIからの報告、orchestratorの受理判断が状態に残っていない場合、その作業は完了扱いにしない、という考え方です。

### user-side tasks

AI側だけで進められない作業は、ユーザー側タスクとして扱います。

たとえば、ユーザーの承認、方針確認、ローカル環境での操作、権限の許可などです。
こうしたものをチャットの中に埋もれさせず、ダッシュボード上で見えるようにします。

### vision / failure / user liaison layers

Orquestaには、プロジェクトを進めるための補助レイヤーがあります。

- vision layer: ユーザーの回答や好みを、採用前の方向性候補として整理する
- failure concierge: 環境エラーや繰り返す失敗を、修理カードのように整理する
- user liaison: ユーザーが答えるべきこと、承認すべきこと、待っていることを整理する

これらは常時監視エージェントではありません。
必要な条件が出たときに起こす、event-drivenな役割です。

### trigger audit と question-candidate visibility

最近入れた仕組みとして、trigger auditがあります。

これは、`vision-curator` などのfoundation roleを常時 watcher にするのではなく、「いま起こすべき条件があるか」を監査結果として見えるようにするものです。

たとえば専門AIのレポートに、ユーザーへ聞いた方がよい質問候補が含まれている場合、それをすぐ生の質問として表示するのではなく、question candidateとして扱います。

trigger auditは、pending question candidates が一定数ある、高優先度である、受理前に確認すべきである、といった条件を見えるようにします。

ただし、trigger audit自体は `vision-curator` を勝手に起動しません。
raw candidateを質問として昇格もしません。
orchestratorが、起こすか、保留するか、別のルートにするかを判断します。

## まだbetaです

Orquestaはまだbeta品質です。

できることは増えていますが、安定版でも、完成品でも、本番運用に耐えると断言できるものでもありません。

今の弱いところは、たとえば次のあたりです。

- もっと多くのクリーンプロジェクトで試す必要がある
- Codex環境によって、使えるスレッド操作やツールが違う
- ダッシュボードや状態ファイルの仕様はまだ変わる可能性が高い
- 公開ドキュメントと実運用がズレないよう、継続して確認が必要
- 日本語・英語の説明、導入記事、デモがまだ足りない

現時点では、安定した完成品というより、実際に使いながら運用ルールとUIを固めている段階です。

## どう試すか

インストール対象はこのリポジトリの `orquesta/` です。

https://github.com/ugin-man/orquesta/tree/main/orquesta

手元で試す場合は、新しいCodexスレッドを対象プロジェクトで開き、次のように依頼します。

```text
Use the orquesta skill to set up this project.
```

日本語なら、たとえばこうです。

```text
orquestaスキルでこのプロジェクトを初期セットアップしてください。
```

入口の説明は `START_HERE.md` にまとめています。

https://github.com/ugin-man/orquesta/blob/main/START_HERE.md

## ほしいフィードバック

今ほしいのは、実際に使ったときの引っかかりです。

- インストールできたか
- 最初の説明で何をすればいいか分かったか
- ダッシュボードに実際のチーム状態が見えたか
- 専門AIへの委任が本当に起きたか
- ユーザー側タスクや承認待ちが分かりやすかったか
- 質問候補や障害整理が、開発の判断に役立つか
- どこが重い、怖い、分かりにくいと感じたか

特に、Codexで大きめの個人開発、ゲーム制作、AI多めのツール、創作系プロジェクトをやっている人に試してほしいです。

## おわりに

Orquestaは、AIに全部を任せるための仕組みではありません。

作りたいのは、Codex上で長期専門AIチームを動かしながら、ユーザーが状況を理解し、質問に答え、判断し、品質を上げていける開発ワークフローです。

まだ荒いです。
ただ、ひとつの巨大なAIチャットに全部を背負わせる方法には限界があると感じています。

同じ課題感があれば、触って、壊して、意見をください。
